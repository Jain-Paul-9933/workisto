# ADR 0001 — Auth: sessions now, scoped JWT at the FastAPI boundary later

- **Status:** Accepted (2026-07)
- **Context increment:** 3 (Auth & accounts API)

## Context

Workisto is a **modular monolith**. Django owns the domain and the
transactional writes; a separate **FastAPI** service will later own a couple of
read-heavy, high-QPS endpoints (availability search, booking-ingest) against the
**same** Postgres. Clients are **three Next.js portals** (customer / provider /
admin) against one role-scoped API. Real-time chat runs on **Django Channels**
(WebSockets). It is a marketplace handling **payments and PII**.

The obvious default — "SPA, therefore JWT" — is wrong here, because three facts
about *this* stack invert it.

## Decision

Use **Django server-side sessions** (httpOnly cookie) as the primary browser
auth. When the FastAPI read service lands, Django will **mint a short-lived
(60–300 s) signed JWT** for that single service hop, which FastAPI verifies
statelessly with a shared key. No long-lived JWTs, no refresh tokens, no
browser-held tokens.

## Why not JWT for the browser session

1. **Next.js is a BFF, not a bare SPA.** The browser holds the cookie; the
   Next.js server proxies to Django. Nothing sensitive sits in `localStorage`,
   so the classic JWT-XSS-token-theft surface simply doesn't exist.
2. **Channels reads the session cookie on the WS handshake for free.** Browsers
   can't set an `Authorization` header on a WebSocket; they *do* send cookies.
   `AuthMiddlewareStack` populates `scope["user"]` out of the box. JWT-over-WS
   needs query-string or subprotocol hacks.
3. **A marketplace needs instant revocation** — ban a bad actor, log out
   everywhere, invalidate on password change. Sessions revoke with a `DELETE`.
   JWTs need a blocklist, which re-introduces the per-request stateful lookup
   that JWT was supposed to remove.

## Why JWT is still right at the FastAPI boundary

For FastAPI to validate a Django *session*, it would have to couple to Django's
session store and format. A short-lived signed **token** is verified with a
shared key — zero coupling, no round-trip — and the short TTL sidesteps the
revocation problem entirely. We adopt statelessness **exactly where the
architecture needs it, and nowhere else**.

## Consequences

- Session backend: **`cached_db`** — Redis read cache + Postgres write-through.
  Fast, shared across Railway instances, durable across a Redis restart.
- CSRF is enforced by DRF `SessionAuthentication` for authenticated unsafe
  requests, and handled at the Next.js BFF layer for server-side calls.
- Self-service signup is limited to `CUSTOMER`/`PROVIDER`; `ADMIN` only via
  `createsuperuser`.
- **Revisit if** a native mobile app (no cookie/BFF layer) joins the roadmap —
  that's the one condition that flips the primary auth toward tokens.
